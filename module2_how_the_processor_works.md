## Before You Begin

Module 1 established how data is represented in memory — bits, bytes, integers, floats, structs, and their physical layout. This module answers the next question: what does the processor actually *do* with that data?

Understanding the processor is not about memorizing instruction mnemonics. It is about building a mental model of the machine that explains why code behaves the way it does — why a function call has overhead, why some loops are fast and others are not, why sorting an array before processing it can double performance. These are not mysteries to be accepted on faith. They are direct consequences of how the processor works, and by the end of this module you will be able to reason about them from first principles.

You will read assembly code generated by the Zig compiler. Not write it — read it. The ability to read assembly is one of the most valuable skills a systems programmer can have. When something is fast or slow, correct or incorrect, the assembly is the ground truth.

## Part 1: The Processor and Its Registers

### 1.1 What the Processor Does

At the most fundamental level, a processor is a machine that repeatedly executes one instruction, then the next, then the next. Each instruction is a small, fixed operation: move data from one place to another, add two numbers, compare two values, jump to a different location if a condition holds. That is all. The sophistication of everything a computer does — from rendering video to training neural networks — emerges from executing billions of these simple operations per second.

The processor has three main components:

**The Arithmetic Logic Unit (ALU)** performs computations: addition, subtraction, bitwise operations, comparisons. It takes inputs from registers or memory and writes results to registers or memory.

**Registers** are the processor's working memory — a small number of named storage locations inside the chip itself, each capable of holding one machine word (64 bits on a 64-bit processor). Registers are orders of magnitude faster than any level of the cache hierarchy. Operations on registers take one clock cycle; operations on main memory take two hundred.

**The Control Unit** orchestrates execution: it fetches instructions from memory, decodes them (determines what operation to perform), and directs the ALU and registers accordingly. It maintains the **program counter (PC)** — also called the instruction pointer — a register that holds the address of the next instruction to execute.

### 1.2 The x86-64 Register File

x86-64 has sixteen general-purpose registers, each 64 bits wide. They have names inherited from decades of x86 history:

| 64-bit | 32-bit | 16-bit | Low 8-bit | Purpose |
|--------|--------|--------|-----------|---------|
| `rax`  | `eax`  | `ax`   | `al`      | Accumulator; return value |
| `rbx`  | `ebx`  | `bx`   | `bl`      | Callee-saved |
| `rcx`  | `ecx`  | `cx`   | `cl`      | 4th argument; counter |
| `rdx`  | `edx`  | `dx`   | `dl`      | 3rd argument |
| `rsi`  | `esi`  | `si`   | `sil`     | 2nd argument |
| `rdi`  | `edi`  | `di`   | `dil`     | 1st argument |
| `rbp`  | `ebp`  | `bp`   | `bpl`     | Base pointer (frame pointer) |
| `rsp`  | `esp`  | `sp`   | `spl`     | Stack pointer |
| `r8`   | `r8d`  | `r8w`  | `r8b`     | 5th argument |
| `r9`   | `r9d`  | `r9w`  | `r9b`     | 6th argument |
| `r10`  | `r10d` | `r10w` | `r10b`    | Caller-saved |
| `r11`  | `r11d` | `r11w` | `r11b`    | Caller-saved |
| `r12`  | `r12d` | `r12w` | `r12b`    | Callee-saved |
| `r13`  | `r13d` | `r13w` | `r13b`    | Callee-saved |
| `r14`  | `r14d` | `r14w` | `r14b`    | Callee-saved |
| `r15`  | `r15d` | `r15w` | `r15b`    | Callee-saved |
| `rip`  | —      | —      | —         | Instruction pointer (read-only) |
| `rflags`| —     | —      | —         | Status flags |

Each 64-bit register can also be accessed in smaller widths. Writing to `eax` (the 32-bit version of `rax`) clears the upper 32 bits of `rax` — an important detail that compilers exploit. Writing to `ax` (the 16-bit version) or `al` (the 8-bit version) leaves the upper bits unchanged, which can cause subtle bugs if not handled carefully.

The **instruction pointer** (`rip`) holds the address of the next instruction. It advances automatically after each instruction. Branch instructions modify `rip` to redirect execution.

The **flags register** (`rflags`) contains status bits set by arithmetic operations: the zero flag (ZF, set when a result is zero), the carry flag (CF, set when an operation produces a carry out of the most significant bit), the sign flag (SF, set when a result is negative), and the overflow flag (OF, set when a signed arithmetic operation overflows). Conditional branches test these flags.

### 1.3 Special-Purpose Registers

**`rsp` — Stack Pointer:** Points to the top of the current stack. The stack grows downward: pushing data decrements `rsp`, popping data increments it. Every function call uses the stack; `rsp` is updated on every call and return.

**`rbp` — Base Pointer (Frame Pointer):** Optionally used to point to the base of the current stack frame, making it easy to refer to local variables and function arguments by fixed offsets from `rbp`. Optimized builds often omit the frame pointer (`-fomit-frame-pointer`) and use `rsp` directly, which frees `rbp` as a general-purpose register.

**`rip` — Instruction Pointer:** Not directly writable by general instructions; it advances automatically. Jump and call instructions set `rip` to the target address.

## Part 2: The Instruction Set

### 2.1 Reading Assembly

Assembly language is a human-readable representation of machine code. Each assembly instruction corresponds to one or a few machine code bytes. The Zig compiler (and every other compiler) translates high-level source code into machine code; `objdump` translates that machine code back into assembly for inspection.

Intel syntax (which we use) writes instructions as:

```
MNEMONIC  destination, source
```

AT&T syntax (used by GCC and GDB by default) reverses this. When reading documentation or StackOverflow answers, check which syntax is used.

An operand can be:
- A **register**: `rax`, `rbx`, `r12`, etc.
- An **immediate**: a constant value like `5`, `0xFF`, `-1`
- A **memory reference**: `[rax]` (the value at the address in `rax`), `[rbp - 8]` (8 bytes below the frame pointer), `[rax + rcx*4]` (base + index*scale, a common array access pattern)

### 2.2 Data Movement

**`mov dst, src`** — Copy the value of `src` into `dst`. This is by far the most common instruction.

```asm
mov rax, 42        ; rax = 42
mov rbx, rax       ; rbx = rax (copy register)
mov rax, [rbx]     ; rax = *rbx (load from memory)
mov [rbx], rax     ; *rbx = rax (store to memory)
mov [rbp-8], rax   ; store rax to stack (local variable)
```

**`push src`** — Decrement `rsp` by 8, then write `src` to `[rsp]`:

```asm
push rax    ; equivalent to: sub rsp, 8; mov [rsp], rax
```

**`pop dst`** — Read from `[rsp]` into `dst`, then increment `rsp` by 8:

```asm
pop rax     ; equivalent to: mov rax, [rsp]; add rsp, 8
```

**`lea dst, [expr]`** — Load Effective Address. Computes the address of the memory reference and stores it in `dst`, *without* performing a memory access. Used for address arithmetic:

```asm
lea rax, [rbx + rcx*4]  ; rax = rbx + rcx*4 (address, not value)
lea rax, [rax + rax*2]  ; rax = rax * 3 (multiply by 3 using address arithmetic)
```

`lea` is a surprisingly powerful instruction: it can add up to three values and multiply by 1, 2, 4, or 8 in a single clock cycle.

### 2.3 Arithmetic

**`add dst, src`** — `dst = dst + src`. Sets flags.

**`sub dst, src`** — `dst = dst - src`. Sets flags.

**`imul dst, src`** — Signed multiply: `dst = dst * src`.

**`idiv src`** — Signed divide: divides `rdx:rax` (a 128-bit value formed from `rdx` and `rax`) by `src`. Quotient → `rax`, remainder → `rdx`. The programmer must sign-extend `rax` into `rdx:rax` using `cqo` before dividing.

**`neg dst`** — Negate: `dst = -dst` (two's complement).

**`inc dst`** / **`dec dst`** — Increment/decrement by 1.

**`and dst, src`** / **`or dst, src`** / **`xor dst, src`** / **`not dst`** — Bitwise operations.

**`shl dst, count`** / **`shr dst, count`** / **`sar dst, count`** — Shift left, shift right (logical), shift right arithmetic (preserves sign bit).

A common compiler optimization: multiplication by a constant power of 2 becomes a shift. Multiplication by small constants uses `lea`.

```asm
; Multiply rax by 8:
shl rax, 3      ; rax <<= 3  (equivalent to rax * 8)

; Multiply rax by 5:
lea rax, [rax + rax*4]  ; rax = rax + rax*4 = rax*5
```

### 2.4 Comparison and Flags

**`cmp a, b`** — Computes `a - b` and sets flags, but discards the result. Only the flags matter.

**`test a, b`** — Computes `a & b` and sets flags, discarding the result. `test rax, rax` is the idiom for checking whether `rax` is zero.

The flags set by `cmp` and `test`:
- **ZF (Zero Flag):** Set if result is zero. After `cmp a, b`: set if `a == b`.
- **SF (Sign Flag):** Set if result is negative.
- **OF (Overflow Flag):** Set if signed overflow occurred.
- **CF (Carry Flag):** Set if unsigned carry occurred.

### 2.5 Control Flow

**`jmp target`** — Unconditional jump to `target`. Sets `rip = target`.

**Conditional jumps** check the flags and jump if the condition holds:

| Instruction | Condition | Typical use |
|-------------|-----------|-------------|
| `je` / `jz` | ZF=1 | Jump if equal / zero |
| `jne` / `jnz` | ZF=0 | Jump if not equal / not zero |
| `jl` / `jnge` | SF≠OF | Jump if less (signed) |
| `jle` / `jng` | ZF=1 or SF≠OF | Jump if less or equal (signed) |
| `jg` / `jnle` | ZF=0 and SF=OF | Jump if greater (signed) |
| `jge` / `jnl` | SF=OF | Jump if greater or equal (signed) |
| `jb` / `jnae` | CF=1 | Jump if below (unsigned) |
| `ja` / `jnbe` | CF=0 and ZF=0 | Jump if above (unsigned) |
| `js` | SF=1 | Jump if negative |
| `jns` | SF=0 | Jump if non-negative |

**`call target`** — Call a function. Pushes the return address (the address of the next instruction) onto the stack, then jumps to `target`:

```asm
call target    ; equivalent to: push rip; jmp target
```

**`ret`** — Return from a function. Pops the return address from the stack and jumps to it:

```asm
ret            ; equivalent to: pop rip
```

### 2.6 Reading Zig-Generated Assembly

Let's see what Zig generates for a simple function. Create `src/main.zig`:

```zig
export fn add(a: i64, b: i64) i64 {
    return a + b;
}
```

Compile and disassemble:

```bash
zig build-exe src/main.zig -O ReleaseFast
objdump -d -M intel zig-out/bin/main | grep -A 5 "<add>"
```

You should see something like:

```asm
0000000000401000 <add>:
  401000:  48 8d 04 37   lea rax, [rdi + rsi]
  401004:  c3            ret
```

Three instructions. Let's read them:

- `lea rax, [rdi + rsi]` — The first argument (`a`) is in `rdi` (first integer argument register), the second argument (`b`) is in `rsi` (second argument register). `lea` adds them and puts the result in `rax` (return value register).
- `ret` — Return to the caller.

The compiler has recognized that `a + b` is an address computation and used `lea` instead of `add`. This is a valid optimization: `lea` can compute `rdi + rsi` in one instruction, and the result ends up directly in `rax` rather than needing a separate `mov`.

Now a slightly more complex example:

```zig
export fn max(a: i64, b: i64) i64 {
    if (a > b) return a;
    return b;
}
```

Compile and disassemble. In ReleaseFast mode you might see:

```asm
<max>:
  cmp  rdi, rsi
  mov  rax, rsi
  cmovg rax, rdi
  ret
```

This is elegant. `cmp rdi, rsi` sets the flags by computing `a - b`. `mov rax, rsi` sets `rax = b` (the default return value). `cmovg rax, rdi` is a **conditional move**: it sets `rax = a` only if the "greater than" condition holds (if `a > b`). No branch instruction. The conditional move avoids a branch entirely, which matters for branch prediction (covered in Part 4).

## Part 3: The Call Stack and Function Calls

### 3.1 What Happens When You Call a Function

A function call is one of the most common operations in any program, yet it involves a precise sequence of machine-level actions. Understanding this sequence reveals why recursion has depth limits, what "stack overflow" means at the hardware level, and how debuggers can reconstruct a call stack from raw memory.

The System V AMD64 ABI (the calling convention used on Linux and macOS) defines the rules:

**Argument passing:**
- 1st integer argument → `rdi`
- 2nd integer argument → `rsi`
- 3rd integer argument → `rdx`
- 4th integer argument → `rcx`
- 5th integer argument → `r8`
- 6th integer argument → `r9`
- 7th and subsequent arguments → pushed onto the stack right-to-left

**Return value:** Integer/pointer return values → `rax`. If the return value is two machine words, the high word goes in `rdx`.

**Caller-saved registers:** `rax`, `rcx`, `rdx`, `rsi`, `rdi`, `r8`, `r9`, `r10`, `r11`. The callee may freely modify these. If the caller needs their values preserved across a call, it must save them itself (typically by pushing them to the stack before the call and popping after).

**Callee-saved registers:** `rbx`, `rbp`, `r12`, `r13`, `r14`, `r15`. The callee must preserve these. If the callee uses them, it must save them (push) at the start of the function and restore them (pop) before returning.

**Stack alignment:** Before a `call` instruction, `rsp` must be 16-byte aligned. Since `call` pushes 8 bytes (the return address), `rsp` is 8-byte aligned when the callee starts. Functions that need to call other functions typically subtract 8 from `rsp` (or an amount making the total subtraction a multiple of 16) to restore 16-byte alignment.

### 3.2 Stack Frame Layout

When a function is called, the processor and the function together construct a **stack frame**: a region of the stack containing the function's local variables, saved registers, and other bookkeeping.

A typical stack frame layout (growing downward):

```
Higher addresses
┌─────────────────────────┐
│ ...caller's frame...    │
├─────────────────────────┤
│ 7th+ arguments          │ (if any, pushed by caller)
├─────────────────────────┤
│ Return address          │ ← pushed by CALL instruction
├─────────────────────────┤  ← rbp points here (if frame pointer used)
│ Saved rbp               │ ← pushed by callee PUSH rbp / MOV rbp, rsp
├─────────────────────────┤
│ Saved callee registers  │ (rbx, r12-r15 if used)
├─────────────────────────┤
│ Local variables         │
│ Spilled temporaries     │
├─────────────────────────┤  ← rsp points here (top of stack)
Lower addresses
```

### 3.3 Tracing a Function Call

Let's trace what happens at the machine level when one function calls another.

```zig
fn add(a: i64, b: i64) i64 {
    return a + b;
}

pub fn main() void {
    const result = add(3, 4);
    _ = result;
}
```

Compile in Debug mode (to preserve the frame pointer and disable inlining):

```bash
zig build-exe src/main.zig  # Debug mode by default
objdump -d -M intel zig-out/bin/main | grep -A 20 "<add>"
```

You will see something like:

```asm
<add>:
  push rbp              ; save caller's frame pointer
  mov  rbp, rsp         ; set up our frame pointer
  mov  [rbp-0x8], rdi   ; save argument 'a' to stack
  mov  [rbp-0x10], rsi  ; save argument 'b' to stack
  mov  rax, [rbp-0x8]   ; load 'a'
  add  rax, [rbp-0x10]  ; add 'b'
  pop  rbp              ; restore caller's frame pointer
  ret                   ; return

<main>:
  push rbp
  mov  rbp, rsp
  sub  rsp, 0x10        ; allocate 16 bytes on the stack for local vars
  mov  edi, 0x3         ; 1st argument = 3 (using 32-bit register)
  mov  esi, 0x4         ; 2nd argument = 4
  call <add>            ; push return address, jump to add
  mov  [rbp-0x8], rax   ; store result = 7
  add  rsp, 0x10        ; deallocate stack frame
  pop  rbp
  ret
```

Trace through this step by step:

1. `main` sets up its stack frame
2. `main` puts arguments 3 and 4 in `edi`/`esi` (the 32-bit versions of `rdi`/`rsi` — writing to a 32-bit register zero-extends into the 64-bit register, so `edi = 3` sets `rdi = 3`)
3. `call <add>` pushes the address of the next instruction (the `mov [rbp-0x8], rax` line) onto the stack and jumps to `add`
4. `add` saves `rbp` and sets up its own frame pointer
5. `add` spills the arguments to the stack (Debug mode does this for debuggability; ReleaseFast would keep them in registers)
6. `add` computes the result in `rax`
7. `add` restores `rbp` and executes `ret`, which pops the return address and jumps back to `main`
8. `main` stores the return value from `rax` to a stack slot
9. `main` tears down its stack frame and returns

In ReleaseFast mode, all of this collapses to the three instructions we saw earlier. The compiler inlines the `add` call entirely — there is no actual function call at the machine level.

### 3.4 Recursion and Stack Overflow

Every function call allocates a stack frame. If a function calls itself recursively without a base case — or with a very large depth — the stack eventually runs out of space. The OS has set a limit on the stack size (typically 8 MB on Linux). When `rsp` tries to go below the stack's lower bound, a page fault occurs; the OS catches it and sends SIGSEGV to the process.

```zig
fn infinite_recursion(n: i64) i64 {
    return infinite_recursion(n + 1);
}
```

Each call to `infinite_recursion` creates a small stack frame. With 8 MB of stack and ~16 bytes per frame, you can recurse about 500,000 times before the stack overflows. The exact number depends on frame size.

## Part 4: Pipelining — Making the Machine Fast

### 4.1 The Von Neumann Bottleneck

A naive processor executes one instruction at a time: fetch the instruction, decode it, execute it, write the result. If each stage takes one clock cycle and there are five stages, the processor takes five cycles per instruction. On a 3 GHz processor (3 billion cycles per second), that would be 600 million instructions per second — impressive by human standards, but far slower than modern processors actually achieve.

The solution is **pipelining**: executing multiple instructions simultaneously, each at a different stage. Just as a factory assembly line can have multiple cars in production simultaneously (one being painted, one being assembled, one being tested), the processor can have multiple instructions in different stages of execution at the same time.

### 4.2 The Classic Five-Stage Pipeline

The classic five-stage RISC pipeline divides instruction execution into:

```
Stage 1: IF  — Instruction Fetch
         Reads the instruction from memory (or instruction cache)
         at the address in rip. Increments rip.

Stage 2: ID  — Instruction Decode
         Determines what the instruction does.
         Reads operand values from the register file.

Stage 3: EX  — Execute
         The ALU performs the computation.
         For memory instructions, computes the effective address.

Stage 4: MEM — Memory Access
         Loads from or stores to data cache.
         No-op for non-memory instructions.

Stage 5: WB  — Write Back
         Writes the result to the destination register.
```

With a five-stage pipeline, five instructions can be in progress simultaneously:

```
Cycle:    1    2    3    4    5    6    7    8    9
instr 1:  IF   ID   EX   MEM  WB
instr 2:       IF   ID   EX   MEM  WB
instr 3:            IF   ID   EX   MEM  WB
instr 4:                 IF   ID   EX   MEM  WB
instr 5:                      IF   ID   EX   MEM  WB
```

In steady state, the pipeline completes one instruction per cycle. The pipeline's depth (number of stages) determines the maximum speedup. Modern processors have 14-20 stage pipelines, and can execute multiple instructions per cycle (superscalar execution). An Intel Core i9 can retire over 4 instructions per cycle under ideal conditions.

### 4.3 Pipeline Hazards

The pipeline works perfectly only when each instruction is independent of the previous ones. In real programs, instructions frequently depend on each other. These dependencies create **hazards** — situations where the pipeline cannot proceed at full speed.

#### Data Hazards

A data hazard occurs when an instruction needs a result that has not yet been produced by an earlier instruction in the pipeline.

```asm
add rax, rbx    ; rax = rax + rbx (result available after WB = cycle 5)
mov rcx, rax    ; reads rax — but rax isn't ready until cycle 5!
```

If `mov rcx, rax` reads `rax` in the ID stage (cycle 3 for the second instruction), but `add` doesn't write `rax` until the WB stage (cycle 5), there is a two-cycle gap where the wrong value would be read.

**Solution: Register Forwarding (Bypassing)**

Modern processors implement forwarding paths that route results directly from later pipeline stages back to earlier ones, bypassing the register file. The result of the `add` instruction in the EX stage can be forwarded directly to the EX stage of the next instruction, eliminating the stall in many cases.

When forwarding cannot resolve the hazard (for example, a load instruction whose result isn't available until after the MEM stage), the processor inserts a **pipeline stall** (also called a bubble): the dependent instruction is held back for one cycle, causing a one-cycle delay.

```asm
mov rax, [rbx]   ; load from memory (result available after MEM stage)
add rcx, rax     ; depends on load result: 1-cycle stall even with forwarding
```

Good compilers schedule instructions to avoid load-use stalls by placing an independent instruction between a load and its first use.

#### Control Hazards

A control hazard occurs at branch instructions. When the processor is executing a branch (`je`, `jmp`, etc.) in the EX stage, it does not yet know whether the branch is taken or not — that decision depends on the flags, which are computed in EX. But by this point, the IF and ID stages have already fetched the next two instructions (the ones after the branch in memory). If the branch *is* taken, those instructions are wrong and must be discarded.

This is the **branch penalty**: cycles wasted on instructions that should not have been fetched. In a five-stage pipeline, a taken branch costs 2 wasted cycles. In a 20-stage pipeline where branch resolution happens in stage 15, a misprediction can waste 14 cycles.

**Solution: Branch Prediction**

Modern processors use sophisticated hardware branch predictors to guess whether a branch will be taken before it is resolved. If the prediction is correct, execution continues without penalty. If the prediction is wrong, the incorrectly-fetched instructions must be flushed from the pipeline — a **misprediction penalty** of 15-20 cycles on modern processors.

#### Structural Hazards

A structural hazard occurs when two instructions need the same hardware resource at the same time — for example, two memory instructions trying to use the memory bus simultaneously. Modern processors largely eliminate structural hazards through resource duplication (multiple execution units, separate instruction and data caches).

### 4.4 Branch Prediction in Practice

Modern branch predictors achieve over 95% accuracy on typical code. They do this by maintaining a history of recent branch outcomes and using that history to predict future ones. A loop that executes 100 times will be predicted "taken" on the first 99 iterations and "not taken" on the 100th. The predictor learns this pattern quickly.

What defeats branch prediction? **Data-dependent branches** — branches whose outcome depends on data values that change unpredictably. The classic example is sorting: an unsorted array has random comparisons; a sorted array has predictable ones.

Let's measure this:

```zig
const std = @import("std");

const N = 32768;
var data: [N]i32 = undefined;

fn sum_with_branch(arr: []i32) i64 {
    var total: i64 = 0;
    for (arr) |x| {
        if (x >= 128) total += x;
    }
    return total;
}

pub fn main() !void {
    // Initialize with random data
    var prng = std.rand.DefaultPrng.init(42);
    const rand = prng.random();
    for (&data) |*x| {
        x.* = rand.intRangeAtMost(i32, 0, 255);
    }

    const timer = std.time.Timer;
    var t = try timer.start();

    // Unsorted: branches are unpredictable
    var result: i64 = 0;
    const iterations = 1000;
    for (0..iterations) |_| {
        result +%= sum_with_branch(&data);
    }

    const unsorted_ns = t.lap();
    std.debug.print("unsorted result = {d}\n", .{result});
    std.debug.print("unsorted time   = {d} ns\n", .{unsorted_ns});

    // Sort the data
    std.mem.sort(i32, &data, {}, std.sort.asc(i32));

    result = 0;
    t.reset();
    for (0..iterations) |_| {
        result +%= sum_with_branch(&data);
    }

    const sorted_ns = t.lap();
    std.debug.print("sorted result = {d}\n", .{result});
    std.debug.print("sorted time   = {d} ns\n", .{sorted_ns});

    std.debug.print("\nspeedup: {d:.2}x\n",
        .{@as(f64, @floatFromInt(unsorted_ns)) /
          @as(f64, @floatFromInt(sorted_ns))});
}
```

On a modern processor, the sorted version typically runs 2-4x faster, despite doing identical arithmetic. The only difference is branch predictability.

> **Why?** With unsorted data, roughly half the branches are taken and half are not, in random order. The branch predictor cannot learn the pattern and mispredicts frequently, costing 15-20 cycles per misprediction. With sorted data, the branches first always not-taken (values < 128) then always taken (values ≥ 128) — two patterns the predictor learns perfectly, with only one misprediction at the transition point.

### 4.5 Writing Branch-Friendly Code

Understanding branch prediction changes how you write performance-critical code. Strategies:

**Branchless code:** Replace predictable branches with arithmetic or conditional moves. The compiler often does this automatically (`cmov`), but you can hint at it:

```zig
// Branchy: may mispredict
fn max_branchy(a: i64, b: i64) i64 {
    if (a > b) return a else return b;
}

// Branchless: always the same execution path
fn max_branchless(a: i64, b: i64) i64 {
    // a + ((b - a) & ((b - a) >> 63))
    // This is a bitmask trick; the compiler generates cmov
    return @max(a, b); // Zig's @max typically compiles to cmov
}
```

**Sort data before processing:** As demonstrated above, sorting data that will be processed with data-dependent conditions dramatically improves branch predictability.

**Loop unrolling:** Processing multiple elements per iteration reduces the number of loop-control branches:

```zig
// Standard loop: one branch per element
for (arr) |x| { process(x); }

// Unrolled 4x: one branch per 4 elements
var i: usize = 0;
while (i + 3 < arr.len) : (i += 4) {
    process(arr[i]);
    process(arr[i+1]);
    process(arr[i+2]);
    process(arr[i+3]);
}
// Handle remainder
while (i < arr.len) : (i += 1) process(arr[i]);
```

The compiler often unrolls loops automatically in ReleaseFast mode.

## Part 5: The Memory Hierarchy

### 5.1 The Fundamental Problem

Processors have become dramatically faster over the past forty years. Memory bandwidth and latency have improved much more slowly. This gap — processors that can execute billions of operations per second but must wait hundreds of cycles to fetch data from main memory — is the central performance problem of modern computing.

The solution is a **memory hierarchy**: a series of progressively larger, slower, and cheaper storage levels. The processor uses a small amount of very fast storage (cache) and a larger amount of slower storage (main memory). The cache transparently holds the data that the processor is most likely to need next, exploiting a property of programs called **locality**.

### 5.2 Cache Levels and Their Characteristics

A modern x86-64 processor has three levels of cache between the registers and main memory:

| Level | Typical Size | Latency | Per Core? |
|-------|-------------|---------|-----------|
| Registers | ~1 KB (logical) | 0 cycles | Yes |
| L1 cache | 32-64 KB | 4-5 cycles | Yes |
| L2 cache | 256 KB - 1 MB | 12-14 cycles | Yes |
| L3 cache | 8-64 MB | 30-50 cycles | Shared |
| Main memory (DRAM) | 8-256+ GB | 200-300 cycles | Shared |
| NVMe SSD | terabytes | ~100,000 cycles | Shared |

These numbers are not exact — they vary by processor generation — but the ratios are what matter. L1 cache is roughly **50x faster than main memory**. L3 to main memory is a **4-6x** gap. Missing the L3 cache and hitting main memory is expensive enough to be a primary performance bottleneck in real systems.

### 5.3 Cache Lines and Spatial Locality

Cache does not operate at the byte level. When the processor fetches a byte from memory, it fetches the entire **cache line** containing that byte. A cache line is typically 64 bytes on modern x86-64 processors.

This is the physical mechanism behind **spatial locality**: if you access byte at address X, the cache will also load bytes at addresses X+1 through X+63. If your next access is to X+4 (a nearby address), it is a cache hit — the data is already in the cache from the previous fetch. If your next access is to some random far-away address, it is a cache miss.

The implication for data structure design: **access memory sequentially**, and lay out data so that related items are near each other in memory. An array of structs, where you access many fields of each element sequentially, is cache-friendly. A linked list, where each node could be anywhere in memory, is cache-hostile.

### 5.4 Cache Associativity and Replacement

Each cache has a finite number of slots. When the cache is full and a new cache line must be loaded, something must be evicted. The cache replacement policy determines what gets evicted.

Most caches use a **set-associative** design. The cache is divided into sets, and each address maps to exactly one set (based on address bits). Within a set, the cache can hold N cache lines (where N is the "way count"). A 4-way set-associative L1 cache, for example, can hold 4 cache lines for each set.

When all N ways in the set are occupied and a new line must be loaded, the cache evicts the line that was least recently used (**LRU policy**). This approximates the optimal policy (evict the line that will be needed furthest in the future) because temporal locality means recently-used data is likely to be used again soon.

### 5.5 Measuring Cache Performance

The best way to internalize the memory hierarchy is to measure it directly. The following benchmark demonstrates the dramatic performance difference between sequential and random memory access:

```zig
const std = @import("std");

// Large enough to overflow L3 cache (adjust for your machine)
// Typical L3 cache: 8-32 MB. Use 128 MB to guarantee misses.
const ARRAY_SIZE = 128 * 1024 * 1024 / @sizeOf(u64); // 128 MB of u64

pub fn main() !void {
    var gpa = std.heap.DebugAllocator(.{}){};
    defer _ = gpa.deinit();
    const allocator = gpa.allocator();

    const arr = try allocator.alloc(u64, ARRAY_SIZE);
    defer allocator.free(arr);

    // Initialize with sequential values
    for (arr, 0..) |*x, i| x.* = i;

    const iterations: usize = 10_000_000;
    var timer = try std.time.Timer.start();

    // ---- Sequential access ----
    var sum: u64 = 0;
    var idx: usize = 0;
    timer.reset();
    for (0..iterations) |_| {
        sum +%= arr[idx];
        idx = (idx + 1) % ARRAY_SIZE;
    }
    const sequential_ns = timer.lap();
    std.debug.print("Sequential: {d} ns, sum={d}\n", .{ sequential_ns, sum });

    // ---- Strided access (stride = 16, skipping cache lines) ----
    const stride: usize = 16; // 16 * 8 bytes = 128 bytes = 2 cache lines
    sum = 0;
    idx = 0;
    timer.reset();
    for (0..iterations) |_| {
        sum +%= arr[idx];
        idx = (idx + stride) % ARRAY_SIZE;
    }
    const strided_ns = timer.lap();
    std.debug.print("Strided ({d}): {d} ns, sum={d}\n",
        .{ stride, strided_ns, sum });

    // ---- Random access (defeats prefetcher) ----
    // Create a random permutation array to simulate random access
    var prng = std.rand.DefaultPrng.init(12345);
    const perm = try allocator.alloc(usize, iterations);
    defer allocator.free(perm);
    for (perm) |*p| p.* = prng.random().uintLessThan(usize, ARRAY_SIZE);

    sum = 0;
    timer.reset();
    for (perm) |p| {
        sum +%= arr[p];
    }
    const random_ns = timer.lap();
    std.debug.print("Random:     {d} ns, sum={d}\n", .{ random_ns, sum });

    std.debug.print("\nRatios (sequential = 1.0x):\n", .{});
    const seq_f: f64 = @floatFromInt(sequential_ns);
    std.debug.print("Strided: {d:.1}x slower\n",
        .{@as(f64, @floatFromInt(strided_ns)) / seq_f});
    std.debug.print("Random:  {d:.1}x slower\n",
        .{@as(f64, @floatFromInt(random_ns)) / seq_f});
}
```

Run this with `zig build run -Doptimize=ReleaseFast`. Typical results on a modern machine:

```
Sequential: 45,000,000 ns
Strided:    89,000,000 ns
Random:     2,100,000,000 ns

Ratios (sequential = 1.0x):
Strided: 2.0x slower
Random:  46.7x slower
```

Random access is roughly **50x slower** than sequential access. The bottleneck is main memory latency. Sequential access is fast because the hardware prefetcher detects the access pattern and loads cache lines before they are needed.

### 5.6 The Row-Major vs. Column-Major Benchmark

The canonical demonstration of spatial locality is matrix traversal. A 2D matrix stored in row-major order (rows are contiguous in memory) is fast to traverse row by row, slow to traverse column by column.

```zig
const std = @import("std");
const SIZE = 1024; // 1024x1024 matrix of f32 = 4 MB

var matrix: [SIZE][SIZE]f32 = undefined;

fn sum_row_major() f64 {
    var total: f64 = 0;
    for (0..SIZE) |row| {
        for (0..SIZE) |col| {
            total += matrix[row][col]; // sequential memory access
        }
    }
    return total;
}

fn sum_col_major() f64 {
    var total: f64 = 0;
    for (0..SIZE) |col| {
        for (0..SIZE) |row| {
            total += matrix[row][col]; // stride = SIZE * 4 = 4096 bytes!
        }
    }
    return total;
}

pub fn main() !void {
    // Initialize matrix
    for (0..SIZE) |row| {
        for (0..SIZE) |col| {
            matrix[row][col] = @floatFromInt(row * SIZE + col);
        }
    }

    var timer = try std.time.Timer.start();

    timer.reset();
    const row_result = sum_row_major();
    const row_ns = timer.lap();

    timer.reset();
    const col_result = sum_col_major();
    const col_ns = timer.lap();

    std.debug.print("Row-major: {d} ns (result: {d:.0})\n",
        .{ row_ns, row_result });
    std.debug.print("Col-major: {d} ns (result: {d:.0})\n",
        .{ col_ns, col_result });
    std.debug.print("Ratio: {d:.1}x\n",
        .{@as(f64, @floatFromInt(col_ns)) /
          @as(f64, @floatFromInt(row_ns))});
}
```

Expected results: column-major traversal is typically **5-10x slower**, despite performing identical arithmetic on the same data. Every column access jumps 4096 bytes (one row) forward in memory, loading a new cache line for each element and evicting the previous one immediately.

### 5.7 Cache-Aware Data Structure Design

The performance implications of the cache hierarchy reach into data structure design. A linked list is cache-hostile: each node is a separate heap allocation, potentially anywhere in memory. Traversing a linked list of N elements requires N cache misses.

An array is cache-friendly: elements are contiguous. Traversing an array of N elements requires approximately N / (cache_line_size / element_size) cache misses — typically 8-16x fewer.

The **hot/cold field split** is a technique for improving cache utilization in structs that are frequently iterated:

```zig
// Poor: every field is loaded even when only 'value' is needed in the hot loop
const SlowEntry = struct {
    key: [32]u8,   // 32 bytes - rarely needed
    value: f64,    // 8 bytes - frequently needed in hot loop
    metadata: [24]u8, // 24 bytes - rarely needed
    // Total: 64 bytes = 1 cache line. 56 bytes are wasted in the hot loop.
};

// Better: separate hot and cold data
const HotEntry = struct {
    value: f64,    // 8 bytes
    cold_index: u32, // 4 bytes - index into cold data
    // Can pack 8 entries per cache line
};

const ColdEntry = struct {
    key: [32]u8,
    metadata: [24]u8,
};
```

When processing a million entries and only need the `value` field, the slow version loads 64 MB into cache (one cache line per entry). The hot/cold split loads only 8 MB for the hot data — fitting 8 entries per cache line instead of 1.

### 5.8 Prefetching

The hardware prefetcher automatically detects sequential and strided access patterns and loads cache lines before they are requested. It is highly effective for regular access patterns and largely ineffective for random access.

For cases where the access pattern is known in advance but not detectable by hardware (such as following pointer chains in a data structure), software prefetching gives the programmer explicit control:

```zig
// Zig exposes the prefetch builtin
@prefetch(ptr, .{ .rw = .read, .locality = 3, .cache = .data });
```

`locality` 3 means "keep in L1 cache"; locality 0 means "load to L3 only, will not be reused soon." Prefetching is an advanced optimization; use it only after profiling confirms a cache miss problem and other approaches have been exhausted.

## Part 6: Seeing the Machine — Using objdump

### 6.1 The Full Disassembly Workflow

Disassembly is the systematic process of turning a compiled binary back into assembly. It is an essential skill for performance analysis, debugging, and understanding what the compiler is actually doing.

The basic workflow:

```bash
# Compile with debug symbols preserved (for function names)
zig build-exe src/main.zig -O ReleaseFast

# Full disassembly (can be long)
objdump -d -M intel zig-out/bin/main

# Disassemble a specific function
objdump -d -M intel zig-out/bin/main | grep -A 30 "<function_name>"

# Show source interleaved with assembly (requires debug info)
objdump -d -M intel --source zig-out/bin/main

# Compact: just the instructions, no addresses
objdump -d -M intel zig-out/bin/main | grep "^\s*[0-9a-f]*:" | head -50
```

### 6.2 What to Look For

When inspecting compiler output, you are looking for:

**Inlining:** Did the compiler inline a function call? If a function call disappears in ReleaseFast output but is present in Debug output, it was inlined — the compiler eliminated the call overhead and merged the function body into the caller.

**Loop structure:** How does the compiler represent a loop? Look for a backward jump (a jump to a lower address) — that is typically the loop back-edge.

**Vectorization:** Does the compiler use SIMD instructions (`xmm`, `ymm`, or `zmm` registers; instructions like `vmovdqu`, `vaddps`, `vpxor`)? Vectorized loops process multiple elements simultaneously using SIMD units.

**Register allocation:** How many variables does the function use? If the compiler spills registers to the stack, you will see `mov [rbp - N], reg` stores and `mov reg, [rbp - N]` loads in the middle of computation. Spilling means there were not enough registers for all live values.

### 6.3 Comparing Debug and Release Output

Building the same function in both modes and comparing the output is highly instructive. Let's do this with a sum function:

```zig
pub fn sum_array(arr: []const i64) i64 {
    var total: i64 = 0;
    for (arr) |x| total += x;
    return total;
}
```

**Debug mode (`zig build-exe src/main.zig`):**

```asm
<sum_array>:
  push rbp
  mov  rbp, rsp
  mov  [rbp-0x8],  rdi   ; save arr.ptr
  mov  [rbp-0x10], rsi   ; save arr.len
  mov  qword [rbp-0x18], 0  ; total = 0
  mov  rax, [rbp-0x10]   ; rax = arr.len
  test rax, rax
  je   <end>             ; if len == 0, skip
  xor  ecx, ecx          ; i = 0
<loop>:
  mov  rax, [rbp-0x8]    ; rax = arr.ptr
  mov  rdx, [rax + rcx*8] ; rdx = arr[i]
  add  [rbp-0x18], rdx   ; total += arr[i]
  inc  rcx               ; i++
  cmp  rcx, [rbp-0x10]   ; i < len?
  jb   <loop>
<end>:
  mov  rax, [rbp-0x18]   ; return total
  pop  rbp
  ret
```

Every variable is stored on the stack. The loop accesses memory on every iteration even for the loop counter. This is correct but inefficient.

**ReleaseFast mode (`zig build-exe src/main.zig -O ReleaseFast`):**

```asm
<sum_array>:
  test rsi, rsi        ; if len == 0
  je   <done>
  xor  eax, eax        ; total = 0 (in register)
  xor  ecx, ecx        ; i = 0 (in register)
<loop>:
  add  rax, [rdi + rcx*8]  ; total += arr[i]
  inc  rcx
  cmp  rcx, rsi
  jb   <loop>
<done>:
  ret
```

Half the instructions. All variables stay in registers. No stack accesses in the loop body. The compiler has:
1. Eliminated the frame pointer setup and teardown
2. Kept all variables in registers (no spilling)
3. Removed the redundant loads from the stack

In a real benchmark, this function would be vectorized further — using SIMD to process 4 or 8 elements per iteration. You will see that if you look at the actual output on your machine.

### Implementation Approach

Measuring single-instruction latency is tricky because the processor executes instructions out of order and in parallel. The key technique is to create a **dependency chain** — a sequence of instructions where each depends on the previous result, forcing serial execution:

```zig
// Measuring integer add latency:
// Each add depends on the previous result, so they cannot overlap
var result: i64 = seed;
inline for (0..N) |_| {
    result = result + 1; // dependent on previous
}
```

Compare this to a throughput measurement (no dependency chain):

```zig
// Throughput measurement: multiple independent operations
var a: i64 = seed1;
var b: i64 = seed2;
var c: i64 = seed3;
var d: i64 = seed4;
inline for (0..N/4) |_| {
    a = a + 1; // independent of b, c, d
    b = b + 2; // can execute in parallel with a
    c = c + 3;
    d = d + 4;
}
result = a +% b +% c +% d;
```

The latency measurement reveals how many cycles an instruction takes when the next instruction must wait for its result. The throughput measurement reveals how many independent instances of the instruction the processor can execute per cycle.

### Memory Hierarchy Measurement

To measure cache latency at each level, you need arrays of different sizes:
- Smaller than L1 (e.g., 16 KB): accesses stay in L1
- Larger than L1 but smaller than L2 (e.g., 128 KB): accesses mostly in L2
- Larger than L2 but smaller than L3 (e.g., 4 MB): accesses mostly in L3
- Larger than L3 (e.g., 128 MB): accesses go to main memory

```zig
fn measure_memory_latency(
    arr: []u64,
    iterations: usize,
    timer: *std.time.Timer
) f64 {
    // Create a pointer-chasing chain for accurate latency measurement
    // (Random access pattern that the prefetcher cannot predict)
    // Each element points to the next in a random permutation
    var prng = std.rand.DefaultPrng.init(0);
    const rand = prng.random();

    // Initialize as identity permutation, then shuffle
    for (arr, 0..) |*x, i| x.* = i;
    // Fisher-Yates shuffle
    var n = arr.len;
    while (n > 1) {
        n -= 1;
        const k = rand.uintLessThan(usize, n + 1);
        const tmp = arr[n];
        arr[n] = arr[k];
        arr[k] = tmp;
    }

    // Measure pointer-chasing latency
    timer.reset();
    var idx: u64 = 0;
    for (0..iterations) |_| {
        idx = arr[idx % arr.len];
    }
    const elapsed_ns = timer.lap();

    // Prevent optimization
    std.mem.doNotOptimizeAway(idx);

    const cycles_approx = @as(f64, @floatFromInt(elapsed_ns)) * 3.0 / 1e9;
    return cycles_approx / @as(f64, @floatFromInt(iterations));
}
```

### Branch Prediction Measurement

```zig
fn measure_branch_misprediction(timer: *std.time.Timer) void {
    const N = 10_000_000;
    var data: [256]u8 = undefined;

    // Predictable: sorted data, branch always goes same way
    for (&data, 0..) |*x, i| x.* = @intCast(i);

    timer.reset();
    var sum_pred: i64 = 0;
    for (0..N) |i| {
        if (data[i % 256] >= 128) sum_pred += 1;
    }
    const pred_ns = timer.lap();

    // Unpredictable: random 50/50
    var prng = std.rand.DefaultPrng.init(99);
    for (&data) |*x| x.* = prng.random().int(u8);

    timer.reset();
    var sum_unpred: i64 = 0;
    for (0..N) |i| {
        if (data[i % 256] >= 128) sum_unpred += 1;
    }
    const unpred_ns = timer.lap();

    std.mem.doNotOptimizeAway(sum_pred);
    std.mem.doNotOptimizeAway(sum_unpred);

    std.debug.print("Predictable branch:   {d} ns/iter\n",
        .{pred_ns / N});
    std.debug.print("Unpredictable branch: {d} ns/iter\n",
        .{unpred_ns / N});
}
```

## Summary

This module has built a complete model of the processor from the instruction set down to performance behavior.

**Registers** are the processor's working memory — sixteen 64-bit general-purpose registers on x86-64, plus the instruction pointer and flags. Operations on registers are fast; operations on memory are not.

**The instruction set** is the hardware-software interface. Reading assembly is a skill, not a talent — it requires knowing the instruction categories (move, arithmetic, comparison, control flow) and the calling convention (argument registers, return value register, stack protocol).

**Function calls** have a precise machine-level implementation: arguments in registers, return address on the stack, callee-saved registers saved and restored, stack aligned before every call. Understanding this demystifies debugging, enables reading stack traces, and reveals the cost of function call overhead.

**Pipelining** is why processors are fast: multiple instructions in execution simultaneously. But it creates hazards — data dependencies, branch mispredictions — that cost cycles when they occur. Branch prediction is sophisticated and effective, but data-dependent branches can defeat it, costing 15-20 cycles per misprediction.

**The memory hierarchy** is why memory access patterns matter more than arithmetic. L1 cache is 50x faster than main memory. Sequential access is cache-friendly; random access is not. Cache line size (64 bytes) means spatial locality is free — nearby data comes with the data you request.

**objdump** is your window into what the compiler actually produces. Compare Debug and Release builds, look for inlining and vectorization, understand why the compiler made the choices it did.

## Reference: x86-64 Registers Quick Reference

```
Integer argument passing (System V AMD64 ABI):
  1st: rdi    2nd: rsi    3rd: rdx
  4th: rcx    5th: r8     6th: r9
  7th+: stack (right to left)
  Return value: rax (high bits in rdx if needed)

Caller-saved (callee may modify freely):
  rax, rcx, rdx, rsi, rdi, r8, r9, r10, r11

Callee-saved (callee must preserve):
  rbx, rbp, r12, r13, r14, r15

Special:
  rsp  — stack pointer (always points to top of stack)
  rbp  — frame pointer (optional in optimized code)
  rip  — instruction pointer (not directly writable)
  rflags — status flags (ZF, SF, OF, CF, PF, AF)
```

## Reference: Common Assembly Patterns

```asm
; Check if rax == 0:
test rax, rax
je   <target>

; Return 0/1 based on condition (e.g., equal):
xor  eax, eax    ; eax = 0 (also zeroes rax)
sete al          ; al = 1 if ZF set, 0 otherwise

; Multiply by constant (prefer lea/shift over imul for small constants):
lea  rax, [rdi + rdi*4]   ; rax = rdi * 5
shl  rdi, 3               ; rdi *= 8

; Branch-free absolute value (signed):
mov  rdx, rdi
sar  rdx, 63      ; rdx = 0 if positive, -1 (all ones) if negative
xor  rdi, rdx     ; flip bits if negative
sub  rdi, rdx     ; add 1 if negative (since -1 subtracted = +1 added)
; rdi now contains |rdi|

; Null check before use:
test rdi, rdi
jz   <null_handler>
mov  rax, [rdi]   ; safe to dereference
```
